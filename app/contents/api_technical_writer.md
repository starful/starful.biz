# [完全ガイド] API Technical Writer：年収・将来性・未経験ロードマップ

「APIドキュメントなんて、エンジニアが適当に書けばいいじゃないか」
もし君がそう思っているなら、悪いことは言わない。今すぐこのページを閉じて、別の「楽に稼げそうな仕事」を探すべきだ。

だが、もし君が**「ソフトウェアが世界を食い尽くす現代において、その『接続点（API）』を支配する者が、プロダクトの成否を握る」**という事実に気づいているなら、ようこそ。ここは、言葉でコードを操り、開発者の体験（DX: Developer Experience）を設計する、最も泥臭く、最も知的で、そして最も報われる「API Technical Writer」という戦場だ。

IT業界の華やかな表舞台の裏側で、仕様の矛盾に怒り、ソースコードの海に溺れ、それでもなお「世界で最も美しいドキュメント」を追求するプロフェッショナルのリアルを、現役のエキスパートとして、包み隠さずお伝えしよう。

---

## 導入：API Technical Writerという職業の「光と影」

現代のITビジネスにおいて、API（Application Programming Interface）は単なる「機能」ではない。それは「商品」そのものだ。Stripe、Twilio、SendGrid……。これら数十兆円規模の企業が売っているのは、洗練されたAPIと、それを使いこなすための**「完璧なドキュメント」**である。

### 🌟 光：開発者の「神」になれる瞬間
API Technical Writer（以下、APIライター）の最大のやりがいは、自分の書いた一行が、世界中のエンジニアの「詰まり」を解消する瞬間だ。
「このAPI、どう叩けばいいのか全くわからん」とTwitter（現X）で嘆いていた海外のエンジニアが、君の書いたチュートリアルを読んで「Holy shit, this is so intuitive!（なんて直感的なんだ！）」と呟く。その瞬間、君は国境を越え、技術の架け橋となった実感を味わうだろう。

### 🌑 影：地獄の「板挟み」と「情報のゴミ拾い」
しかし、現実は甘くない。
APIライターの日常は、**「仕様書がない」「コードと挙動が違う」「リリース5分前に破壊的変更が入る」**というカオスとの戦いだ。
開発チームからは「忙しいからコードを読んで察してくれ」と言われ、ビジネスサイドからは「明日までにSDKのガイドを完璧に仕上げろ」と無茶振りをされる。君は、誰も全貌を把握していない巨大なシステムの「情報のゴミ拾い」をしながら、それを磨き上げてダイヤモンドに変えなければならない。

この職種は、単なる「文章書き」ではない。**「エンジニアの言語」と「人間の言語」を繋ぐ、高度な外交官であり、デバッガーなのだ。**

## 💰 リアルな年収相場と、壁を越えるための「残酷な条件」

APIライターの年収は、一般的なライターや編集者とは一線を画す。なぜなら、そこには「技術理解」という高い参入障壁があるからだ。しかし、ただ「書ける」だけでは、一定のラインで確実に頭打ちになる。

| キャリア段階 | 経験年数 | 推定年収 (万円) | 年収の壁を突破するための「リアルな必須条件」 |
| :--- | :--- | :--- | :--- |
| ジュニア | 1-3年 | 450 - 600 | 言われた仕様をMarkdownに落とすだけでなく、実際にcURLでAPIを叩き、レスポンスの矛盾を指摘できるか |
| ミドル | 3-7年 | 650 - 950 | 開発フローに「Docs as Code」を導入し、GitHub Actions等でドキュメント更新を自動化する仕組みを構築できるか |
| シニア/リード | 7年以上 | 1,000 - 1,500+ | 経営層に対し「ドキュメントの質が解約率（Churn Rate）にどう影響するか」を数値で証明し、予算と人員を勝ち取れるか |

### ⚠️ 「年収800万円の壁」の正体
多くのライターが800万円付近で停滞する。その理由は、**「受け身の姿勢」**だ。
開発者が持ってきた情報を整理するだけなら、AI（ChatGPTやClaude）で事足りる。1,000万円を超えるトップ層は、仕様が決まる前のデザインレビューに参加し、「そのエンドポイントの設計は、外部の開発者にとって直感的ではない。こう変えるべきだ」と、**API設計そのものに口を出す。**
君が「ドキュメント係」ではなく「DX（開発者体験）の設計者」になれるか。それが、残酷なまでの格差を生む。

## ⏰ API Technical Writerの「生々しい1日」のスケジュール

キラキラしたカフェでMacを開いて執筆……そんな幻想は捨てろ。APIライターの1日は、探偵のような調査と、泥臭い調整の連続だ。

*   **09:00：Slackの「炎上」チェック**
    昨晩リリースされた新機能について、サポートチームから悲鳴が上がっている。「ドキュメント通りにリクエストを送っても401エラーが出る」という報告だ。
*   **10:00：朝会（スタンドアップミーティング）**
    バックエンドエンジニアたちが「昨日のバグの原因」について専門用語で議論している。君はその会話を盗み聞きしながら、ドキュメントのどの部分を修正すべきか、あるいは「仕様そのものがおかしい」のかを瞬時に判断する。
*   **11:00：ソースコードとの格闘**
    「仕様書は後で書く」と言い残して休暇に入ったエンジニアの代わりに、GitHubでプルリクエストを追いかける。GoやJavaのコードを読み、バリデーションロジックを解読して、ドキュメントに記載すべき「隠れた制約」を見つけ出す。
*   **13:00：ランチ（という名の情報収集）**
    あえてエンジニアと同じテーブルに座り、世間話を装って「次のスプリントで入る破壊的変更」のリーク情報を得る。
*   **14:00：集中執筆タイム（Deep Work）**
    OpenAPI（Swagger）ファイルを編集し、複雑な認証フローをわかりやすい図解に落とし込む。ここで発生する「仕様の矛盾」に気づき、再度エンジニアに突撃する。「ここのフィールド、必須って書いてあるけど、コード上はオプショナルになってますよ？」
*   **16:00：他部署からの「無茶振り」対応**
    マーケティング部門から「新しいキャンペーンのために、APIのユースケースを素人でもわかるように今すぐ書き換えてくれ」という依頼が入る。技術的な正確性を保ちつつ、いかに平易な言葉にするか。脳の別の回路を使う作業だ。
*   **18:00：ドキュメント・デプロイ**
    CI/CDパイプラインを回し、ステージング環境で表示を確認。リンク切れがないか、サンプルコードが実際に動作するかを最終チェック。
*   **19:00：退勤**
    頭はパンパンだが、自分の書いたドキュメントが世界中のシステムを繋ぐ「血流」になることを誇りに思い、帰路につく。

## ⚖️ この仕事の「天国（やりがい）」と「地獄（きつい現実）」

### 🌈 【やりがい】
1.  **「プロダクトの最初のユーザー」という特権**
    新しいAPIが世に出る前、それを世界で最初に触り、評価するのは君だ。君のフィードバック一つで、APIの使い勝手が劇的に改善される。これは、開発者と対等に渡り合えるプロにしか味わえない快感だ。
2.  **「一度書けば、数万人の時間を救える」というレバレッジ**
    君が書いた「よくあるエラーの解決策」が、世界中のエンジニアの合計数万時間を節約する。この圧倒的な社会的インパクトは、他の職種ではなかなか味わえない。
3.  **希少価値による「圧倒的な市場優位性」**
    「コードが読めて、文章が書けて、設計ができる」人間は、市場に絶望的に不足している。一度このスキルを身につければ、不況など恐れるに足りない。

### 🔥 【地獄（きつい現実）】
1.  **「できて当たり前、間違えれば戦犯」の重圧**
    ドキュメントのサンプルコードに一箇所タイポ（誤字）があるだけで、ユーザー企業のシステムは動かなくなる。そのミスが原因で「この会社のAPIは信頼できない」というブランド毀損に繋がる恐怖と、常に隣り合わせだ。
2.  **終わりのない「仕様変更」との追いかけっこ**
    アジャイル開発の現場では、昨日書いたことが今日には嘘になる。常に変化し続けるターゲットを撃ち続けるような徒労感に、メンタルを削られる人間も多い。
3.  **「ただのライター」と侮られる屈辱**
    技術に疎いマネージャーから「ドキュメントなんて、最後にちょちょっと書けばいいんでしょ？」と言われる。その一言に耐え、自分の価値を証明し続けなければならない。

## 🛠️ 現場で戦うための「ガチ」スキルマップと必須ツール

教科書に載っているような「正しい日本語」なんて二の次だ。現場で求められるのは、**「エンジニアの思考をトレースし、彼らの言語で語り、かつ読者を迷わせない」**ための武装だ。

| スキル・ツール名 | 現場での使われ方（「なぜ」必要なのか、具体的なシーン） |
| :--- | :--- |
| **OpenAPI (Swagger)** | APIの構造を定義する世界標準。これを知らないとエンジニアとの会話が成立しない。 |
| **Git / GitHub** | Docs as Codeの基本。エンジニアと同じワークフローでドキュメントを管理するため。 |
| **Postman / cURL** | 実際にAPIを叩くための武器。「動かない」という報告に対し、自ら検証して原因を特定するため。 |
| **Markdown / MDX** | 執筆の標準言語。Reactコンポーネントを埋め込み、動的なドキュメントを作るため。 |
| **英語（読解・執筆）** | 最新の技術仕様は全て英語。また、グローバル展開するAPIなら英語が一次ソースになる。 |
| **交渉力・政治力** | 開発リソースをドキュメント整備に割いてもらうため、エンジニアやPMを説得する武器。 |
| **コードリーディング** | ドキュメントがない暗黒のコードから、真実（仕様）を掘り起こすための考古学的スキル。 |

> **プロの視点：** 
> 「日本語力」よりも「デバッグ力」を磨け。APIライターの仕事の8割は、ドキュメントを書く前の「調査と検証」にある。

## 🎤 激戦必至！API Technical Writerの「ガチ面接対策」と模範解答

面接官は、君が「綺麗な文章を書けるか」なんて見ていない。**「カオスな現場に放り込んで、自力で情報を整理して、リリースまで漕ぎ着けられるか」**を見ている。

### 質問1：「開発者が忙しくて仕様を教えてくれない時、あなたならどうしますか？」
*   **面接官の意図**: 受動的な姿勢か、能動的に動けるかを確認したい。
*   **NGな回答例**: 「上司に相談して、エンジニアから情報を聞き出してもらうようにお願いします。」
*   **評価される模範解答**: 「まず、自分でソースコード（PR）を読み、自分なりの理解でドラフトを作成します。その上で、エンジニアに『ここまでは理解しましたが、このエッジケースの挙動だけ確信が持てません。5分だけ確認させてください』と、彼らの工数を最小限にする形でアプローチします。また、普段から彼らのコードレビューを眺め、信頼関係を築いておきます。」

### 質問2：「ドキュメントの正確性と、リリースのスピード。どちらを優先しますか？」
*   **面接官の意図**: リスク管理能力と、ビジネス判断のバランス感覚を見たい。
*   **NGな回答例**: 「何があっても正確性が第一です。間違った情報は出せません。」（これではビジネスが止まる）
*   **評価される模範解答**: 「基本的には正確性ですが、リリースの重要度によります。致命的な誤解を招く部分は死守しつつ、軽微な補足などは『Coming Soon』としてリリースを優先し、直後に追記する柔軟な対応をとります。ただし、認証周りなどセキュリティに直結する箇所は、100%の正確性が担保されるまでリリースを止めるよう進言します。」

### 質問3：「あなたが書いたドキュメントに対して、ユーザーから『全くわからない』とクレームが来ました。どう対応しますか？」
*   **面接官の意図**: 批判をどう受け止め、改善プロセスに繋げられるか。
*   **NGな回答例**: 「そのユーザーの理解力不足かもしれないので、より丁寧に説明し直します。」
*   **評価される模範解答**: 「まず、そのユーザーがどのステップで、どのような背景（言語、環境）で躓いたのかをヒアリングします。ドキュメントは『正しい』だけでは不十分で、文脈が欠けていることが多いからです。得られたフィードバックを元に、概念図の追加や、クイックスタートガイドの構成を見直し、同様の不満を持つ潜在的なユーザー全員の体験を改善します。」

### 質問4：「APIの破壊的変更（Breaking Changes）をユーザーに伝える際、最も配慮することは？」
*   **面接官の意図**: ユーザー（開発者）への共感能力と、移行コストの理解。
*   **評価される模範解答**: 「単に『変わります』と伝えるだけでなく、**『なぜ変わるのか（メリット）』『いつまでに対応が必要か』『具体的な移行手順（コードの書き換え例）』**の3点をセットで提供します。特に、新旧のコードを並べた比較（Before/After）を提示し、ユーザーの移行コストを最小化することに全力を尽くします。」

### 質問5：「AI（LLM）がドキュメントを自動生成する時代に、人間が書く意味は何だと思いますか？」
*   **面接官の意図**: 自分の職種のコアバリューを理解しているか。
*   **評価される模範解答**: 「AIは『既存のコードから説明を作る』のは得意ですが、『ユーザーが何を達成したいのか（目的）』に基づいたストーリー構成や、複数のAPIを組み合わせた『ベストプラクティス』の提示はまだ困難です。また、仕様の矛盾を見つけ出し、エンジニアと議論して設計を正すという『上流工程のコンサルティング』こそが、人間にしかできない価値だと考えています。」

## 💡 未経験・ジュニアからよくある質問（FAQ）

### Q1. プログラミングスクールを出た程度の知識でなれますか？
**A. 正直に言おう、それだけでは「お話にならない」。**
スクールで習う「アプリの作り方」と、APIライターに求められる「システムの裏側の理解」は別物だ。HTTPステータスコードの意味、認証（OAuth2.0など）の仕組み、JSONの構造などを、空で説明できるレベルまで叩き込む必要がある。ただし、スクールで「コードへの抵抗感」を無くしたことは大きな一歩だ。次は、OSSのドキュメント修正にプルリクエストを送ることから始めろ。

### Q2. 数学の知識はどこまで必要ですか？
**A. ほとんどの場合、高度な数学は不要だ。**
それよりも必要なのは「論理的思考（ロジカルシンキング）」だ。AならばB、BならばC、という連鎖を、一分の隙もなく説明できる能力。そして、複雑なものを「抽象化」して整理する能力だ。数学ができることよりも、パズルを解くのが好きだったり、プラモデルの説明書を読み込むのが好きだったりする性格の方が向いている。

### Q3. 英語ができないと致命的ですか？
**A. 致命的だ。ただし、ペラペラである必要はない。**
APIの世界の公用語は英語だ。最新のライブラリのドキュメント、Stack Overflowの回答、GitHubのIssue……全て英語だ。英語が読めないということは、情報の一次ソースにアクセスできないということであり、それはプロとして致命的なハンデになる。DeepLやChatGPTを使いこなすのは良いが、最終的なニュアンスの正誤を判断できる「読解力」は必須だ。

### Q4. 30代・40代からの未経験挑戦は可能ですか？
**A. むしろ、大歓迎されるケースもある。**
もし君に、前職での「特定業界の深い知識（ドメイン知識）」があるなら、それは強力な武器になる。例えば、金融業界にいた人がフィンテック企業のAPIライターになる場合、技術的なことは後から学べても、金融の複雑な商習慣や法規制を理解していることは、若手エンジニアには真似できない価値になる。

### Q5. AIに仕事を奪われませんか？
**A. 「言われたことを書くだけのライター」は、あと2年で消えるだろう。**
だが、この記事で何度も書いたように、エンジニアと喧嘩し、仕様の矛盾を突き、ユーザーの体験を設計する「APIストラテジスト」としての仕事は、むしろ需要が増している。AIは「嘘（ハルシネーション）」をつくが、APIドキュメントにおいて嘘は許されない。その「最後の砦」として責任を取れるのは、人間だけだ。
